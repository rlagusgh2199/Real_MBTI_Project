# Real MBTI 프로젝트 Constitution (프로젝트 원칙)

## 1. 프로젝트 목적

- 설문이 아닌 **행동 데이터(카톡, 유튜브, SNS 등)**를 기반으로
  사용자의 실제 성향을 분석하는 "Real MBTI" 시스템을 만든다.
- 생성형 AI(LLM)를 활용하여 사용자가 이해하기 쉬운 **자연어 리포트**를 자동으로 만들어준다.

## 2. 핵심 원칙

1. **데이터 최소 수집**
   - 과제/실습 단계에서는 민감한 개인정보를 실제로 수집하지 않고,
     예시 데이터 또는 사용자가 직접 제공한 작은 샘플로 진행한다.
   - 실제 서비스를 가정할 때도 "필요한 최소 데이터만 사용"을 기본 원칙으로 한다.

2. **투명한 분석 과정**
   - MBTI 결과가 어떻게 나왔는지 사용자가 대략 이해할 수 있도록,
     "어떤 데이터 → 어떤 특징 → 어떤 성향 점수"인지 설명을 남긴다.
   - LLM 리포트에도 가능한 한 분석 근거를 포함한다.

3. **설명 가능한 결과 (Explainability)**
   - "당신은 E가 높습니다."에서 끝나는 것이 아니라,
     "이런 대화 패턴/시청 패턴 때문에 E가 높게 나왔습니다." 형태로 설명을 제공한다.

4. **재현 가능한 파이프라인**
   - 데이터 전처리 → 특징 추출 → 점수 계산 → LLM 리포트 생성의 흐름이
     코드와 문서에 명확히 드러나도록 설계한다.
   - 입력이 같으면 항상 같은 점수 결과가 나오도록 한다. (LLM 리포트 문장은 달라져도 됨)

5. **AI 남용 금지**
   - LLM에게 모든 것을 맡기지 않고,
     **규칙 기반/점수 계산 로직 + LLM 요약/설명 역할**로 역할을 분리한다.
   - 성향 점수 자체는 되도록 규칙·수식 기반으로 계산하고,
     LLM은 "설명, 요약, 자연어 리포트"에 집중시킨다.

## 3. 사용자 경험(UX) 원칙

1. **한눈에 이해 가능한 결과 화면**
   - MBTI 4축 점수(E/I, S/N, T/F, J/P)를 그래프나 표로 보여준다.
   - 텍스트로는 "요약 → 상세 설명" 순서로 보여준다.

2. **부담 없는 입력 방식**
   - 실제 카카오톡 전체 백업이 아니라,
     예시로 몇 개의 대화/댓글/텍스트를 붙여 넣는 방식으로도 동작하게 설계한다.

3. **피드백 가능성**
   - 사용자가 "나는 이 성향이 아닌 것 같다"라고 느낄 경우,
     어떤 데이터를 추가로 넣으면 결과가 달라질 수 있는지 설명해 줄 수 있도록 설계한다.
   - (예: "업무 관련 대화가 더 많이 포함되면 판단이 바뀔 수 있습니다." 등의 메시지)

## 4. 기술적인 가이드라인

1. **언어 및 스택**
   - 백엔드/데이터 파이프라인: Python
   - LLM 연동: API (예: OpenAI, 기타 LLM API) 또는 Local LLM (예: Ollama, LM Studio 등)
   - 간단한 웹 UI 또는 CLI 중 하나를 선택 (시간에 따라 조정)

2. **모듈화**
   - `data_loader` : 텍스트 데이터 로딩/정제
   - `feature_extractor` : 감정 분석, 키워드 추출, 행동 패턴 분석
   - `mbti_scorer` : MBTI 4축 점수 계산
   - `llm_reporter` : LLM 호출 → 리포트 생성
   - `app` : 전체 플로우를 실행하는 엔트리 포인트

3. **테스트**
   - 최소한 MBTI 점수 계산 함수는 단위 테스트를 만든다.
   - 같은 입력에 대해 항상 동일한 점수가 나오는지 확인한다.

## 5. 범위(Scope)

- 이 프로젝트는 **수업 과제**를 목적으로 하며, 실제 서비스 상용화를 목표로 하지 않는다.
- 민감한 개인정보는 다루지 않으며, 실제 데이터 대신
  "샘플/익명화된/가짜 데이터"를 사용해도 괜찮다.
